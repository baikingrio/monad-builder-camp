# Safe SDK 学习笔记：从 Smart Account 到多签交易流程

- 学习日期：2026-07-18
- 主资料：[Safe SDK Overview](https://docs.safe.global/sdk/overview)
- 学习目标：理解 Safe SDK 各 Kit 的职责，并能判断一个 Safe 应用应从哪一层开始接入。

## 1. Safe 解决什么问题

普通 EOA 钱包由一把私钥直接控制：签名有效，交易通常就能执行。

Safe 则是一个运行在链上的智能合约账户（Smart Account）。账户的控制规则写在合约里，例如：

- 哪些地址是 owner；
- 至少需要几位 owner 确认（threshold）；
- 是否启用了 Module 或 Guard；
- 一笔交易是否先提案、等待多签，再由任意人执行。

因此，Safe 的重点不只是“多签钱包”，而是把账户权限、交易协作和可扩展执行规则放到合约中。

## 2. Safe SDK 的定位

Safe 官方将 SDK 定位为接入 Safe 智能账户的开发工具集合：它把部署和操作智能合约账户的复杂性封装起来，也能接入不同第三方服务。

SDK 总览把能力分成四个 Kit：

```text
应用前端 / 服务端
        │
        ├─ Starter Kit：高层、统一的起点
        ├─ Protocol Kit：直接操作 Safe 合约能力
        ├─ API Kit：与 Transaction Service 协作
        └─ Relay Kit：ERC-4337 / UserOperation 等执行路径
```

它们不是完全互斥的产品。Starter Kit 会封装其他 Kit 的复杂逻辑；当项目有更具体的需求时，开发者再选择下沉到对应的 Kit。

## 3. 四个 Kit 分别负责什么

### 3.1 Starter Kit：默认从这里开始

包名：`@safe-global/sdk-starter-kit`

Starter Kit 是官方推荐的 TypeScript 起点。它提供较简化、可组合的接口，用来与 Safe Smart Account 交互。

官方总览列出的典型交易形式包括：

- 多签交易（multi-signature transactions）；
- UserOperation；
- 链下和链上消息（off-chain / on-chain messages）。

我的理解：如果只是要在一个应用里接入 Safe 创建和常见交易流程，应先看 Starter Kit。不要一开始就把所有底层合约和服务接口都拼起来。

### 3.2 Protocol Kit：直接操作 Safe 合约

包名：`@safe-global/protocol-kit`

Protocol Kit 面向需要更细控制的场景，可通过 TypeScript 与 Safe Smart Account 直接交互。官方文档列出的能力包括：

- 创建或部署新的 Safe；
- 更新已有 Safe 的配置；
- 创建、提案和执行交易；
- 处理交易签名和消息签名；
- 多链 Safe 部署。

可以把它理解为：**应用需要明确操控 Safe 本身的链上规则时，用 Protocol Kit。**

例如，产品要做一个团队金库管理页面，允许管理员新增 owner、调整 threshold，或者构造一笔批量调用交易，就需要理解这一层的对象和交易生命周期。

### 3.3 API Kit：让多签成员围绕交易协作

包名：`@safe-global/api-kit`

API Kit 连接 Safe Transaction Service API。它不等于直接执行合约调用，而是补足多签协作所需的服务层能力：

- 提案并分享一笔 Safe 交易；
- 将 signer 的签名提交给服务，收集确认；
- 查询交易历史和待确认交易；
- 查询 Safe 信息，例如启用的 Modules、Guards 等。

我的理解：Protocol Kit 偏“如何构造和执行 Safe 交易”；API Kit 偏“这笔交易怎样被成员看到、确认和跟踪”。

### 3.4 Relay Kit：面向 ERC-4337 的执行路径

Relay Kit 面向账户抽象的交易执行，文档中包含 ERC-4337 Safe SDK 和 Gelato Relay 等路径。

在 ERC-4337 模型中，用户可能提交的是 `UserOperation`，而不是传统 EOA 直接广播的交易。Bundler、Paymaster 或 relay 服务会参与后续处理，因此可以支持更灵活的 Gas 支付与交易体验。

这里要避免一个常见误解：

> Gas 代付不等于绕过账户授权。

即使交易由 Paymaster 或 relay 支付 Gas，Safe 的 owner / threshold / Module / Guard 等授权规则仍然需要被正确满足。

## 4. 一笔典型 Safe 多签交易的流程

下面是我目前对标准多签流程的简化理解：

```text
1. 构造交易
   目标地址、value、calldata、operation 等
        ↓
2. 创建 Safe Transaction
   用 Protocol Kit 或 Starter Kit 生成可签名的 Safe 交易
        ↓
3. 提案与共享
   可通过 API Kit 提交到 Transaction Service
        ↓
4. Owner 分别确认
   每位 owner 对同一笔交易签名
        ↓
5. 达到 threshold
   例如 2/3 Safe 收集到两份有效签名
        ↓
6. 执行
   Safe 合约验证签名和规则后，执行目标调用
```

这套流程的价值是：资金或关键权限的操作不依赖单点私钥；每笔敏感操作可被提案、确认和审计。

## 5. Safe、ERC-4337 与 Monad 学习的关系

Safe 是一种智能合约账户，因此它与 Account Abstraction 的概念有天然联系，但两者不能简单画等号：

- Safe 的核心是智能合约账户与可配置授权规则；
- ERC-4337 是一套让智能账户通过 `UserOperation` 进入链上执行的标准化基础设施路径；
- Safe 可以在传统多签交易流程中工作，也可以结合 ERC-4337 / relay 能力改善 Gas 与交互体验。

对 Monad Builder Camp 的后续实践来说，一个合理的学习顺序是：

1. 先理解 owner、threshold、Safe Transaction 和多签确认；
2. 再用 Safe SDK 构造一个小型 Safe 交易流程；
3. 最后研究 UserOperation、Paymaster 和 relay 如何改变提交与 Gas 支付体验。

在没有明确授权和安全设计之前，不应该把私钥、服务端凭证、Paymaster 密钥或自动执行权限放进浏览器，也不应该把“能代付 Gas”误写成“可以自动绕过多签”。

## 6. 我今天形成的判断

- 新项目接入 Safe：优先从 **Starter Kit** 开始；
- 需要部署、改 owner / threshold、构造或执行 Safe 交易：研究 **Protocol Kit**；
- 需要交易提案、签名收集、待确认队列和历史查询：接入 **API Kit**；
- 需要 ERC-4337、UserOperation 或 relay：再学习 **Relay Kit**。

Safe SDK 的关键不只是“调用一个钱包 API”，而是把智能账户、链上授权规则和多人交易协作拆成了清晰的层次。

## 7. 下一步实践题

下一次可以做一个只读 / 测试网练习：

1. 用 Starter Kit 连接一个测试 Safe；
2. 读取 Safe 地址、owner 列表和 threshold；
3. 构造一笔小额或零 value 的测试交易；
4. 只在确认 Safe、网络、签名者和执行权限都正确后，再决定是否真实广播；
5. 记录交易哈希、Safe 地址、链、签名数量与 explorer 链接作为 Proof of Work。
