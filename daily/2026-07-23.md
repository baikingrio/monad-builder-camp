# 2026-07-23｜Week 3 Day 4：把 Session Key 权限边界落到可检查的证据

## 今日目标

今天继续完善 Monad Lucky Draw 的账户抽象实践，不把“免弹窗抽卡”停留在页面体验层，而是核查 Session Key 的链上权限边界：它到底能调用什么、不能调用什么，以及缺失的策略依赖是否真的可以在 Monad Testnet 使用。

## 今天完成了什么

### 1. 锁定 Safe 7579 与 Smart Sessions 的上游来源

我为后续的 7579 迁移建立了可检查的依赖清单，固定了：

- Safe 7579 v2：`rhinestonewtf/safe7579` 的 `v2.0.0` / commit `40d92be...`；
- Smart Sessions：`erc7579/smartsessions` 的 `v1.0.0` / commit `dc20f0...`；
- 对应 Solidity source、artifact、creation bytecode 和 runtime bytecode 的哈希；
- Lucky Draw 允许动作：目标合约 `0x4b3c1adBeeb0776ee31Fd51Eb6169da97A222E70`、`draw()`、selector `0x0eecae21`、`value = 0`。

我没有复制不明来源的合约，也没有把浮动 npm 版本或跨链地址当作 Monad 可用依赖。

### 2. 确认 Monad Testnet 不能直接复用现成 7579 基础设施

此前以只读 `eth_getCode` 检查了 Safe 7579 Launchpad、Adapter、Smart Session Emissary、Universal Action / Time Frame / Value Limit Policy 的常用地址；它们在 Monad Testnet 都没有 bytecode。

这意味着不能只把其他链的固定地址写进前端或脚本。后续若要走 7579 路径，必须先有完整的部署来源、代码哈希与部署前验证。

### 3. 建立“默认禁止部署”的验证 gate

今天新增了两层本地检查：

- Foundry manifest 测试固定上游 commit、artifact 哈希、Lucky Draw target / selector / zero value，以及 Monad 上缺失的依赖地址；
- `npm test` 会先检查所有依赖是否为精确版本，并检查 Safe 7579 manifest 是否仍处于禁止部署状态。

当前 manifest 明确要求以下 blocker 必须存在且保持未解除：

- `timeFramePolicyAudited`；
- `valueLimitPolicyAudited`；
- `fullPolicyStackPinned`；
- `monadCodeVerified`。

也就是说，不能因为页面已经能跑就跳过 expiry、zero value 或部署字节码的链上验证。

### 4. 查清当前上游策略为什么还不能安全落地

我继续核查了已发布的 Smart Sessions 策略：

- Smart Session 原生 `ActionData` 可以收紧为指定 Lucky Draw 合约和 `draw()` selector；
- Owner 可以通过官方机制撤销 Session；
- 但 Session 本身没有原生 expiry 字段；
- 已发布的 `UniActionPolicy` 虽含 `valueLimitPerUse`，但要求至少一条 calldata 参数规则；Lucky Draw 的 `draw()` 没有参数，因此不能把它直接当作“无参数抽卡的零 value 策略”；
- `TimeFramePolicy` 和 `ValueLimitPolicy` 在当前固定的 v1.0.0 源码里只出现在 `test/mock/`，没有可接受的 production artifact 或部署溯源。

因此，今天的正确结论不是“已经完成更安全的 Session Key”，而是：**当前 7579 迁移仍被 expiry / zero-value 的可审计策略依赖阻断。**

## 关键理解

Session Key 的安全不等于“服务端限制一下调用次数”。真正需要分层确认：

```text
用户授权
→ 链上指定 target / selector
→ value 与有效期限制
→ 可撤销机制
→ UserOperation 发送前验证
→ 链上回执与可复查证据
```

其中任何一层缺失，都不能把“免弹窗”描述为完全受限或适合扩大使用范围。对于今天这个无参数 `draw()` 场景，不能用 mock Policy 或服务端 TTL 假装替代链上 expiry / zero-value 约束。

## 残酷共学打卡草稿

今天我继续完善 Monad Lucky Draw 的账户抽象实践，重点不是再加一个“免弹窗”按钮，而是核查 Session Key 的真实权限边界。

我把 Safe 7579 和 Smart Sessions 的上游版本、合约 artifact 哈希、允许调用的 Lucky Draw 地址、`draw()` selector 和 `value = 0` 都固定成可检查的 manifest，并增加了测试 gate：依赖版本不是精确版本、关键策略依赖缺失，或有人误把部署状态改成可用，验证都会失败。

今天最大的收获是：Session Key 的安全不能只靠服务端限流。指定合约和指定方法、零 value、有效期、撤销机制和 UserOperation 验证都应该有链上或可复查的边界。当前我确认了一个重要限制：Smart Sessions 的 TimeFrame / ValueLimit Policy 在固定上游版本中只有 mock 来源，不能直接用于真实部署；而已发布的 UniActionPolicy 也不适配无参数的 `draw()` 调用。

所以今天没有为了“跑通”而接入未经审查的策略，也没有创建新 Safe 或广播交易。下一步会继续寻找有审计与部署溯源的策略；在此之前，旧的测试网 Demo 边界保持不变。
