# 2026-07-19｜Safe Passkeys 与 Monad Safe 4337 实践

## 今日目标

今天原本想按 Safe 官方 Nuxt Passkeys 教程，把 Passkey 创建、Safe 部署和 NFT Mint 的流程跑一遍；后面又专门确认了这条路线能不能直接放到 Monad Testnet。

## 今天学习 / 完成了什么

我先按 Safe 的 Nuxt Passkeys 教程，在 `experiments/erc4337-frontend` 里补了一个 `/safe-passkeys` 页面。

这个页面保留在 Ethereum Sepolia：浏览器可以创建 Passkey，公开的 `rawId` 和公钥坐标只存在本机浏览器的 localStorage；只有手动点击 Mint 按钮时，才会请求 Passkey 签名并提交 UserOperation。没有配置 Pimlico key 时，页面不会尝试部署或发送交易。

然后我查了 Monad 对账户抽象和 Safe 的支持，结论比一开始想得更清楚：

- Monad Testnet 已部署 ERC-4337 EntryPoint v0.6、v0.7、v0.8；
- Safe 官方把 Monad Testnet 列为 Safe Smart Account、Safe 4337 Module、Safe{Core} SDK、Safe{Wallet}、Transaction Service 和 Event Service 的支持网络；
- 但是 Safe 官方网络目录没有把 **Safe Passkey Module** 标成 Monad Testnet 的支持项。

所以不能只把 Sepolia 教程的 RPC 换成 Monad，就说 Safe Passkeys 在 Monad 上已经是官方支持方案。比较稳妥的拆法是：Sepolia 继续做官方 Passkeys 教程；Monad 单独做标准 Safe 4337 路线。

基于这个判断，我新增了 `/safe-4337-monad` 页面，先做只读的链上基础验证。页面会读取 Monad Testnet 上三份固定合约的 bytecode：

- Safe v1.4.1：`0x41675C099F32341bf84BFc5382aF534df5C7461a`
- Safe 4337 Module：`0x75cf11467937ce3F2f357CE24ffc3DBF8fD5c226`
- EntryPoint v0.7：`0x0000000071727De22E5E9d8BAf0edAc6f37da032`

今天实际点过页面里的验证按钮，三项都从 Monad Testnet 返回了链上代码。这个步骤不连接钱包、不创建 Safe、不签名，也不发送 UserOperation。

## 关键理解

今天最大的收获是把三个概念分开了：

1. **Safe** 是智能合约账户和权限规则；
2. **ERC-4337** 是通过 UserOperation、Bundler、Paymaster 执行智能账户操作的基础设施；
3. **Passkey** 是一种认证和签名体验，但是否能和某条链上的 Safe 官方模块直接组合，要看模块部署与支持情况，不能只看链上有 EntryPoint。

对 Monad 来说，当前可直接实践的路径是 Safe 1.4.1 + Safe 4337 Module + EntryPoint v0.7，再接 Pimlico 的 Bundler / Paymaster。Safe Passkey 则需要作为自部署 signer / verifier 的进阶路线处理。
