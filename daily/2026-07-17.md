# 2026-07-17｜Week 2：从学习 Moss 到尝试参与核心架构贡献

通过梳理依赖关系，我确认 #65 不是一个独立的小功能，它依赖：

```text
#61 Bound Protocol construction
  ↓
#62 Trusted / Package Receipt label renderer
  ↓
#63 OnChain metadata observation cache
  ↓
#65 ERC-721 collection binding + metadata
```

其中 #61 已有维护者正在推进的 Draft PR，但尚未合并；#62 没有 blocker，是目前最适合参与的前置核心任务。

## 选择的贡献方向：Issue #62

我计划尝试参与：

- [#62 Core: render Trusted and Package labels in Receipts](https://github.com/nishuzumi/moss/issues/62)

这个任务的目标不是修改链上交易本身，而是在 Registry 层为 Receipt 提供统一、可验证的地址展示机制。

我理解的核心边界包括：

- Trusted token catalog 必须由 composition root 显式提供；
- Protocol 可以声明自己的固定 Package address label；
- 标签优先级需要固定，不能因注册顺序而变化；
- 只允许替换 Receipt 中面向用户展示的文本；
- Outcome、Change、hash、calldata 等结构化证据不能被改写；
- OnChain metadata cache 属于后续 #63，不应混入 #62；
- ERC-20、ERC-721 的具体迁移属于 #64、#65，也不应提前抢做。

这让我更清楚地理解了 Moss 的设计：可读性可以增强，但不能反过来改变交易证据。展示层和证据层必须分开。

## 今日学习总结

今天最大的收获是：参与开源不只是“找到一个 Issue 然后开始写代码”。

对于 Moss 这种强调安全边界和证据完整性的项目，先确认当前 main、现有 PR、Issue 依赖和责任边界，比快速写出一段代码更重要。特别是在核心架构演进期间，避免重复实现、避免把后续任务提前混进来，本身也是高质量贡献的一部分。

我希望后续不只是完成一次 PR，而是沿着 #62 → #63 → #65 这条链路，逐步理解 Moss 如何把协议身份、链上 metadata、Receipt 展示和原始执行证据分层处理。
