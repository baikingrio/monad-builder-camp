## 1. Monad 对用户来说熟悉在哪里？

Monad 官方文档里提到，Monad 是一个高性能、兼容 Ethereum 的 L1。它对用户和开发者来说有两个关键词：

- **portability**：可迁移性
- **performance**：性能

可迁移性体现在：

- Monad 兼容 EVM bytecode。
- Monad 兼容 Ethereum RPC。
- 以太坊上的应用可以更容易迁移到 Monad。
- 用户仍然可以使用熟悉的钱包和地址体系。
- 开发者仍然可以使用 Solidity、EVM 工具链、区块浏览器和现有基础设施。

这说明 Monad 不是让用户重新学习一套完全不同的链，而是在尽量保留以太坊生态开发和使用习惯的基础上，提升底层性能。

我的理解是：

> Monad 的第一层体验差异，是“熟悉但更快”。用户不用抛弃原来的钱包、地址和 EVM 认知，开发者也不用从零学习一套完全不同的虚拟机。

## 2. Monad 的性能体验来自哪里？

官方文档提到，Monad 目标支持：

- 10,000 TPS
- 400ms block frequency
- 800ms finality

这些数字说明 Monad 不只是想做一个“稍微快一点的 EVM 链”，而是希望让链上应用可以承载更高频、更连续的用户交互。

但我觉得更重要的是理解这些性能背后的设计，而不是只记住数字。

Monad 的优化包括：

- MonadBFT：更高性能的 BFT 共识机制
- RaptorCast：更高效的区块传播
- Asynchronous Execution：异步执行，把共识和执行进行流水线化
- Parallel Execution：并行执行，提高交易处理效率
- JIT Compilation：提升 EVM 执行性能
- MonadDb：优化 Ethereum 状态访问和存储

传统 EVM 链的瓶颈包括：

- 存储访问效率不高
- 单线程执行限制吞吐
- 共识和执行耦合，导致执行预算有限
- 状态增长会影响后续访问成本

Monad 的思路不是简单扩大某一个参数，而是从共识、执行、状态存储、网络传播等多个环节一起优化。

我的理解是：

> Monad 的“快”不是一个单点功能，而是整套链架构对 EVM 性能瓶颈的系统性优化。

## 3. 为什么 Monad 体验不只是 TPS 更高？

如果只说 Monad TPS 高，其实还不够。

TPS 高意味着链可以处理更多交易，但用户真正感受到的体验还包括：

- 钱包等待时间是否更短
- 交易确认是否更快
- 应用是否可以支持连续操作
- 多步骤交互是否不再卡顿
- 链游、社交、任务、积分、交易等场景是否可以更顺滑

例如在低性能链上，一个应用如果每次点击都要等待较长确认时间，就很难做成接近普通互联网产品的体验。用户会觉得“每一步都在等链”。

但如果链可以承载更高吞吐、更短区块时间和更快 finality，产品设计就可以从“低频交易”扩展到“连续交互”。

比如：

- 链游可以有更多实时动作。
- DeFi 可以支持更高频的报价、下单和组合操作。
- 社交或任务系统可以把更多行为记录上链。
- 积分、徽章、活动、签到类产品可以更自然地链上化。
- 多步骤合约交互可以变得更接近普通 App 流程。

所以 Monad 的体验差异不只是“交易更快”，而是让开发者可以重新思考哪些产品交互可以放到链上。

## 4. Monad 与 Ethereum 的相同点和不同点

### 相同点

Monad 保留了很多 Ethereum 开发者熟悉的部分：

- EVM 兼容
- Ethereum RPC 兼容
- 地址体系类似
- 可以使用熟悉的钱包
- 可以使用 Solidity 和现有 EVM 工具链
- 应用迁移成本相对较低

这对生态建设很重要，因为开发者不需要完全更换技术栈。

### 不同点

Monad 也不是完全复制 Ethereum。开发者需要注意一些差异：

- 合约 code size 上限更高：Monad 最高 128 KB，而 Ethereum 是 24 KB。
- Monad 交易按 **gas limit** 收费，而不是按实际 gas used 收费。
- 交易处理和执行机制与异步执行有关。
- 没有全局 mempool，交易会转发给后续几个 leader。
- 高吞吐下 full node 不适合提供任意历史状态访问，因此应用更需要 indexer。

这些差异说明：

> Monad 对开发者友好，但开发者不能完全照搬 Ethereum 的习惯。要想发挥 Monad 的性能优势，需要理解它自己的执行、gas 和数据读取模型。

## 5. Gas Pricing 带来的开发启发

Monad 官方文档中有一个很重要的点：

> 在 Monad 中，交易按 gas limit 收费，而不是按实际 gas used 收费。

这是为了支持异步执行并防止 DoS。

如果按照 gas used 收费，用户可以提交 gas limit 很高但实际消耗很低的交易，占用区块 gas 空间却支付很少费用，这会成为 DoS 风险。

因此在 Monad 上，开发者需要更认真地处理 gas limit。

官方建议：

- 如果某个操作 gas 使用量固定，可以显式设置 gas limit。
- 不要每次都依赖 `eth_estimateGas`。
- 显式设置 gas limit 可以减少钱包等待 RPC 的时间。
- 如果合约调用会 revert，一些钱包可能会设置过高 gas limit，这在 Monad 上会带来更差体验。

我的理解是：

> Monad 的高性能不是只靠链底层完成，应用开发者也要配合更好的交易设计，才能让用户真正感受到快。

## 6. 高性能应用的开发实践

Monad 的 Best Practices 文档里提到，高性能应用不应该只关注合约，还要关注前端、RPC、数据读取和交易发送方式。

几个重要实践：

- 对固定 gas 的操作，显式设置 gas limit。
- 多个 `eth_call` 不要串行请求，可以并发请求。
- 可以使用 Multicall 或 batch RPC 减少往返延迟。
- 对 read-heavy 应用，应该使用 indexer，而不是频繁调用 `eth_getLogs`。
- 如果短时间发送多笔交易，可以本地管理 nonce，并发提交交易。

这让我意识到：

> 高性能链上的应用，也必须用高性能方式开发。否则即使链底层很快，应用层仍然可能被 RPC、前端请求、数据索引和钱包交互拖慢。

## 7. 我对“为什么 Monad 体验不同”的总结

我现在的理解是：Monad 的体验不同主要体现在三个层面。

### 第一，用户体验：熟悉但更顺滑

用户仍然可以使用熟悉的钱包、地址和 EVM 应用逻辑，但链的吞吐和确认速度更适合连续操作。

这会降低用户迁移成本，也让链上应用更接近普通互联网产品体验。

### 第二，开发者体验：兼容但需要新习惯

开发者可以沿用 Solidity、EVM 工具、RPC 和区块浏览器，但也要理解 Monad 的差异，例如 gas limit 收费、异步执行、高吞吐下的数据索引和并发请求设计。

这不是“完全换一条技术栈”，而是“在熟悉的 EVM 基础上，用更高性能的方式开发”。

### 第三，产品体验：从低频交易走向高频交互

过去很多链上应用更像“提交交易工具”，用户每一步都要等待确认。Monad 这样的高性能链让开发者有机会设计更多高频、连续、实时的产品体验。

比如链游、实时交易、社交任务、积分系统、多步骤交互应用，都可能因为更低延迟和更高吞吐而变得更自然。

## 9. ERC-4337 最小实践补充

在完成 Monad 理解任务之后，我决定再补一个 Monad Testnet 上的 ERC-4337 最小实践，因为前一天读了 ERC-4337 和 Session Keys，如果只停留在文档理解还不够。

这次实践目标不是做生产级 AA 钱包，而是把 ERC-4337 的最小链上路径跑通：

1. 部署一个最小智能账户 `Minimal4337Account`。

### 10.1 实践中保持不变的部分

从这次最小迁移看，v0.7 和 v0.8 在最基础的 ERC-4337 路径上是连续的：

1. 仍然是先构造 `PackedUserOperation`。
2. 仍然通过 EntryPoint 的 `getUserOpHash` 计算待签名 hash。
3. 仍然由账户 owner 对 UserOpHash 签名。
4. 仍然调用 EntryPoint 的 `handleOps`。
5. EntryPoint 仍然会回调账户的 `validateUserOp`。
6. 验签通过后，账户仍然执行自己的 `execute` 逻辑。

所以对我这个最小 demo 来说，迁移到 v0.8 的第一步并不复杂。它证明了之前写的最小账户模型没有被推翻，`UserOperation -> EntryPoint -> validateUserOp -> execute` 这条核心路径依然成立。

### 10.2 实践中实际变化的部分

这次代码层面最大的变化是 EntryPoint 地址：

```text
v0.7: 0x0000000071727De22E5E9d8BAf0edAc6f37da032
v0.8: 0x4337084D9E255Ff0702461CF8895CE9E3b5Ff108
```

我新建了脚本：

```text
experiments/monad-playground/script/Minimal4337PracticeV08.s.sol
```

脚本里仍然复用 `Minimal4337Account`，但构造账户时传入 v0.8 EntryPoint。这样账户的 `entryPoint` immutable 字段会固定为 v0.8 地址，后续只有这个 EntryPoint 能调用 `validateUserOp`。

这一步也让我意识到：AA 账户不是只看账户合约本身，还要看它绑定的是哪个 EntryPoint。换 EntryPoint 后，最好重新部署账户，而不是假设旧账户可以自动适配新入口。

### 10.3 v0.8 比 v0.7 更重要的升级点

从 ERC-4337 v0.8 release notes 看，v0.8 不只是“换一个地址”，它还补了几个更面向实际产品的问题：

1. **支持 EIP-7702**
   v0.8 EntryPoint 原生支持 EIP-7702 授权，UserOperation hash 会包含 7702 delegation address，也引入了 `eip7702Auth` 相关能力。这说明 ERC-4337 正在和 EOA 临时变成智能账户的方向结合。

2. **支持 ERC-712 风格的 UserOperation hash / signature**
   这对硬件钱包、外部签名器和更标准的签名体验更友好。

3. **修复 initCode front-running 问题**
   旧版本里，UserOperation 里的 `initCode` 有可能被别人提前拿去执行，导致账户提前被部署，原来的 UserOperation 失败。v0.8 引入 `SenderCreator` 相关改动，Factory 推荐检查：

   ```solidity
   require(msg.sender == entryPoint.senderCreator());
   ```

   这对我下一步做 counterfactual deployment 很重要，因为下一步正是要用 `initCode` 在 UserOperation 里顺便创建账户。

4. **改进 paymaster 相关问题**
   v0.8 修复了 Paymaster `postOp` 里 `actualGasUsed` 计算的问题，也禁止显式把 paymaster 设成零地址但又带 paymaster data。这说明 paymaster 虽然能做 gasless/sponsored UX，但边界和安全细节非常多，不能随便写一个“看起来能跑”的版本就当生产可用。

5. **unused gas penalty 更细**
   v0.7 引入 unused gas penalty，v0.8 调整为 unused gas 低于 40,000 时不再惩罚。这对开发者设置 gas limit 有帮助，也提醒我在 Monad 上做 AA 交互时不能随便把 gas limit 设得特别大。

6. **暴露 `senderCreator()`**
   v0.8 把 SenderCreator 地址公开出来，方便 factory、测试和账户部署流程做更清楚的校验。

7. **错误处理更细**
   例如 undeployed sender 和 `validateUserOp` revert 会被拆成更明确的错误，不再都混在同一种错误里。对调试 AA 流程很有帮助。

### 10.4 我自己的理解

这次迁移让我对 v0.7 和 v0.8 的关系有了更具体的认识：

- v0.7 已经能跑通最核心的 AA 流程。
- v0.8 更像是把 AA 往真实应用推进了一步，尤其是 EIP-7702、initCode 安全、paymaster 计费和错误处理。
- 如果只是最小 `handleOps` demo，v0.8 的迁移成本不高。
- 但如果要做真正有用户体验价值的 AA 产品，比如 session key、gas sponsorship、counterfactual account、小游戏自动操作、任务积分自动领取，那么 v0.8 的升级点就会变得很重要。

我觉得这也和 Monad 的方向很契合。Monad 提供的是高吞吐、低延迟、EVM 兼容的基础环境，而 ERC-4337 v0.8 提供的是更好的账户抽象入口。两者结合后，更适合探索高频交互产品：比如任务系统、排行榜、小游戏、链上社交或 AI Agent 代操作。用户不一定每一步都用主钱包手动签名，但关键授权、账户行为和积分变化仍然可以留下链上记录。

下一步我准备继续做 `initCode` / counterfactual deployment：让账户不是提前部署，而是在第一次 UserOperation 里通过 factory 创建出来。这样会更接近真实 AA 钱包的 onboarding 体验。

我选择 Week 2 继续走 **Tech Track**。原因是我本身有全栈和 Web3 开发经验，这周也已经跑通了 Monad Testnet 上从合约源码、Foundry 测试、部署、源码验证、合约交互到 ERC-4337 最小账户实践的完整路径。相比只做概念学习，我更希望后续把这些基础能力推进成一个可以展示的链上 Demo，比如围绕高频交互、任务系统、积分、排行榜、小游戏或 AA 钱包体验做一个更完整的 Monad 应用原型。

本周最重要的几个收获：

1. 我对链上产品和普通互联网产品的区别有了更实际的理解。普通互联网产品的状态主要保存在平台数据库里，用户只能相信平台展示；链上产品则可以把关键交易、合约和状态变化公开记录下来，用户可以通过区块浏览器验证。

2. 我完整走通了 Monad Testnet 的基础开发路径。包括 Foundry 项目搭建、合约编写、测试、部署、源码验证、write function 交互，以及 ERC-4337 的最小 UserOperation 流程。

3. 我对 Monad 的理解不再停留在“更快”这个词上。Monad 的价值不只是高 TPS，而是通过 EVM 兼容、高吞吐、低延迟和更高性能的执行 / 存储 / 共识设计，让链上应用有机会支持更连续的用户交互。比如链游、任务系统、积分、排行榜、实时社交、多步骤合约交互和 AA 钱包，这些场景如果链很慢或者手续费很高，用户体验会很差。

2. 在 Monad 上做高频交互 Demo 时，哪些状态更适合上链，哪些状态应该留在普通数据库或前端缓存中？我希望后续 Demo 不只是“把所有东西都上链”，而是真的利用链上公开验证、低延迟和 EVM 兼容的优势。
