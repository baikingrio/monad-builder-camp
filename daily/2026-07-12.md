# 2026-07-12｜ERC-7702 真实委托与 Sponsor 授权学习

## 今天完成了什么

今天先继续整理 ERC-4337 Sponsor 授权的安全边界，并完成了一次真实的 ERC-7702 Monad Testnet 实践。

Sponsor 部分，我保留了“后端只签短时授权、默认不公开”的原则。完整 UserOperation 不变时，Paymaster 的只读校验返回 `validationData = 0`；只改动 `gasFees` 后会返回 `validationData = 1`。这让我更明确：Sponsor 不是替用户做授权，而是对一笔具体请求是否愿意承担 Gas 做风控判断。

ERC-7702 部分，我阅读了 EIP-7702、Ethereum Pectra 资料和 Monad 官方文档，并在 Monad Testnet 上完成了真实的 `type 0x04` 交易。

这次没有只停留在本地模拟：课程 EOA 签署了指向受限 Executor 的委托授权，独立 relayer 提交交易并支付 Gas。交易执行后，EOA 的 code 变成 delegation indicator，指向 Executor；Executor 再以该 EOA 的上下文对固定 Target 连续执行两次 `checkIn()`。

## 今天最大的收获

最大的收获是：ERC-7702 并不是“给 EOA 加一个万能执行器”这么简单。真正困难的部分是授权边界和 Gas 责任。

Monad 有一条很重要的规则：已委托 EOA 如果因为交易扣减余额后低于 `10 MON`，交易会被拒绝。课程 EOA 当前不适合在委托后继续自己支付 Gas，所以这次把“签授权”和“付 Gas”拆开：EOA 负责签链绑定授权，relayer 负责提交并支付 Gas。

但 relayer 不能因此获得通用账户控制权。因此我没有给 Executor 设计任意 target 或任意 calldata 的入口，而是把它限制为：只接受固定 relayer、只调用固定 Target、只执行两次零价值 `checkIn()`。这次实践让我理解到，做链上账户体验时，功能越强，越要先把“谁能调用、能调用什么、花谁的钱”写成可验证的合约约束。
