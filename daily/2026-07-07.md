# 2026-07-07｜ERC-4337 与 Session Keys 学习笔记

## 今日学习主题

今天阅读了两份资料，主要围绕两个主题：

- **ERC-4337 / Account Abstraction**：账户抽象，让钱包从 EOA 变成可编程账户
- **Session Keys**：会话密钥，让某个临时/受限密钥代表主账户执行操作

阅读资料：

- ERC-4337 Docs：https://docs.erc4337.io/index.html
- thirdweb Engine Session Keys Guide：https://portal.thirdweb.com/engine/v3/guides/session-keys

可以把它们合起来理解为：

> ERC-4337 提供“智能账户 + UserOperation + Bundler + EntryPoint + Paymaster”的底层账户抽象框架；Session Keys 则是在智能账户之上实现更好 UX 的授权模式，让用户不必每次都用主钱包签名。

---

## 1. ERC-4337 是什么？

ERC-4337 是以太坊上的 **Account Abstraction，账户抽象** 标准。

它的目标是：

> 不修改以太坊底层协议，也能让钱包具备智能合约能力。

传统以太坊账户主要有两类：

- **EOA**：Externally Owned Account，外部拥有账户，比如 MetaMask 钱包地址，由私钥控制。
- **Contract Account**：合约账户，由代码控制。

传统模式下，用户发交易必须依赖 EOA 签名，并且 EOA 本身功能很弱：

- 只能用私钥签名
- 必须有 ETH 付 gas
- 无法内置权限管理
- 无法原生支持社交恢复、多签、Session Key、批量交易等复杂逻辑

ERC-4337 的思路是：

> 让用户的钱包变成智能合约账户，签名验证、权限控制、gas 支付方式都可以由合约逻辑自定义。

---

## 2. ERC-4337 的核心价值

ERC-4337 解锁了很多更接近普通互联网产品体验的钱包功能：

- **自定义签名方式**
  - 不一定只能用 EOA 私钥
  - 可以支持 passkey、多签、社交恢复、guardian 等

- **Gasless Transaction**
  - 用户不一定需要自己持有 ETH
  - 可以由 Paymaster 代付 gas

- **批量操作**
  - 多个链上操作可以打包成一次用户行为
  - 例如 approve + swap 一键完成

- **首次使用时创建账户**
  - 用户不一定需要提前部署钱包合约
  - 可以通过 `initCode` 在第一次操作时部署智能账户

- **模块化账户**
  - 权限、恢复、Session Key、插件等能力可以模块化扩展

- **更好的新用户体验**
  - 用户可以先开始使用，再逐步理解链上账户细节

## 3. ERC-4337 的核心组件

### 3.1 UserOperation

`UserOperation` 是 ERC-4337 里最核心的数据结构。

它不是普通以太坊交易，而是一个表达“用户意图”的对象。

可以理解为：

> 用户不再直接发一笔 transaction，而是提交一个 UserOperation，描述自己想做什么。

文档里的结构大致包括：

```ts
interface UserOperation {
  sender: address;
  nonce: uint256;
  initCode: bytes;
  callData: bytes;
  callGasLimit: uint256;
  verificationGasLimit: uint256;
  preVerificationGas: uint256;
  maxFeePerGas: uint256;
  maxPriorityFeePerGas: uint256;
  paymasterAndData: bytes;
  signature: bytes;
}
```

关键字段理解：

- `sender`：智能账户地址
- `nonce`：防重放
- `initCode`：如果账户还没部署，可以用它来部署账户
- `callData`：真正要执行的操作内容
- `paymasterAndData`：如果使用 Paymaster 代付 gas，这里会包含相关数据
- `signature`：用户或授权者的签名，具体怎么验证可以由智能账户自己定义

---

### 3.2 Alt Mempool / UserOp Mempool

普通交易进入以太坊的交易池。

而 ERC-4337 的 `UserOperation` 会进入一个独立的 **UserOp mempool / alt-mempool**。

也就是说：

> ERC-4337 没有改以太坊底层交易池，而是在应用层引入了一套新的 UserOperation 流程。

---

### 3.3 Bundler

Bundler 可以理解为 ERC-4337 世界里的“打包者”。

它的工作是：

1. 监听 UserOp mempool
2. 选择一些 UserOperation
3. 本地模拟验证这些 UserOperation 是否有效
4. 把多个 UserOperation 打包成一笔链上交易
5. 调用 EntryPoint 的 `handleOps()`

Bundler 自己需要先垫付链上交易 gas，因此它必须提前模拟，避免被恶意 UserOperation 消耗 gas。

所以 Bundler 关心：

- 签名是否有效
- nonce 是否正确
- 账户是否有钱或 Paymaster 是否愿意付费
- 操作是否符合 ERC-7562 等验证规则
- Paymaster 是否可能恶意拖垮系统

---

### 3.4 EntryPoint

EntryPoint 是 ERC-4337 的核心入口合约。

可以理解为：

> 所有 UserOperation 最终都通过 EntryPoint 统一验证和执行。

EntryPoint 主要负责：

- 调用智能账户的 `validateUserOp()`
- 如果有 Paymaster，调用 `validatePaymasterUserOp()`
- 验证通过后执行 `callData`
- 管理账户、Paymaster 的 deposit / stake
- 通过 `handleOps()` 执行 Bundler 打包好的 UserOperations

智能账户通常会检查：

```solidity
msg.sender == EntryPoint
```

确保操作确实是通过可信 EntryPoint 触发，而不是被别人随便调用。

---

### 3.5 Paymaster

Paymaster 是可选组件，用来帮用户付 gas。

可以理解为：

> Paymaster 是一个愿意在特定条件下替用户支付 gas 的合约。

常见用途：

- 新用户 gasless onboarding
- dApp 补贴用户操作
- 用 ERC-20 支付 gas
- 游戏、社交应用里免 gas 操作
- 根据 API 或白名单判断是否赞助

Paymaster 要承担风险：

- 如果验证通过但执行失败，Paymaster 仍然可能要支付网络费用
- 因此 Paymaster 通常需要 stake 和 deposit
- 也需要设计防滥用规则

## 4. ERC-4337 的执行流程

可以用一条线理解：

```text
用户/应用生成 UserOperation
        ↓
发送给 Bundler
        ↓
Bundler 本地模拟验证
        ↓
Bundler 打包多个 UserOperation
        ↓
调用 EntryPoint.handleOps()
        ↓
EntryPoint 调用智能账户 validateUserOp()
        ↓
如有 Paymaster，调用 validatePaymasterUserOp()
        ↓
验证通过后执行 callData
        ↓
账户或 Paymaster 支付 gas
```

重点是：

> ERC-4337 把“验证”和“执行”分开了。验证阶段决定这笔 UserOperation 能不能被执行；执行阶段才真正调用目标合约。

## 5. Session Keys 是什么？

Session Key 可以翻译为 **会话密钥**。

它的核心作用是：

> 让一个临时或受限的密钥，在一定权限范围内代表主账户执行操作，而不需要暴露主账户私钥，也不需要用户每次都手动签名。

传统钱包体验里，用户每做一次链上操作都要弹窗签名：

- 玩链游，每走一步都签名，很糟糕
- 高频交互，每个动作都签名，很麻烦
- 移动端 dApp，每次唤起钱包体验差
- 连续应用流程，每一步都签名会打断用户体验

Session Key 的目标是改善这些问题。

比如：

```text
用户主钱包授权一个 Session Key：
- 有效期 1 天
- 只能调用某个游戏合约
- 每次最多花费 0.01 ETH
- 只能操作特定函数
```

之后 dApp 或后端可以用这个 Session Key 在权限范围内执行操作。

---

## 6. Session Key 与 ERC-4337 的关系

ERC-4337 本身没有强制规定 Session Key 标准。

更准确地说：

> ERC-4337 提供了智能账户和 `validateUserOp()` 的机制，Session Key 是钱包、模块或插件在这个机制之上实现的一种授权模式。

在 ERC-4337 智能账户中，可以在 `validateUserOp()` 阶段检查：

- 这笔操作是不是由 Session Key 签的
- Session Key 是否仍在有效期内
- 目标合约是否被允许
- 调用函数是否被允许
- 金额是否超限
- nonce 是否正确
- 是否需要 revoke

所以，Session Key 是账户抽象带来的典型 UX 能力之一。

---

## 7. Session Key 的两种授权模型

资料里提到两类 delegation model。

### 7.1 Static Delegation

静态授权。

特点：

- 给某个 key 持续权限
- 权限会一直存在，直到主动 revoke
- 适合长期设备、后台服务、固定权限场景

风险：

- 如果忘记撤销，权限会长期存在
- 如果 Session Key 泄露，攻击窗口较大

---

### 7.2 Dynamic Delegation

动态授权。

特点：

- 用户签一个带约束条件的 session token
- 约束可以包括：
  - `validUntil`
  - `targetContract`
  - function selector
  - spending limit
  - allowed recipient
- 在 `validateUserOp()` 里由钱包逻辑验证

优点：

- 更灵活
- 更安全
- 可以短期授权
- 更适合临时会话

注意：

> Dynamic Delegation 不是 ERC-4337 原生标准能力，而是钱包自己通过插件、模块或自定义验证逻辑实现。

## 8. thirdweb Session Keys Guide：ERC-4337 智能账户版本

thirdweb 的文档给了一个具体实现路径。

### 前置条件

需要：

- 已配置 thirdweb client
- 一个 session key account address
- Engine 操作用的 Vault access token

### 步骤 1：给用户 Smart Wallet 添加 Session Key

在 ERC-4337 智能账户模式下，第一步是：

> 把 Session Key 地址添加为用户智能账户的 signer，并配置权限。

React 应用里可以通过：

- `ConnectButton`
- `ConnectEmbed`

配置：

```tsx
<ConnectButton
  accountAbstraction={{
    chain: sepolia,
    sessionKey: {
      address: sessionKeyAccountAddress,
      permissions: {
        approvedTargets: "*",
      },
    },
    sponsorGas: true,
  }}
/>;
```

这里的核心字段：

- `sessionKey.address`：被授权的 Session Key 地址
- `permissions.approvedTargets`：Session Key 可以调用哪些目标合约
  - `"*"` 表示允许所有目标
  - 生产环境不推荐直接用 `"*"`，最好限制为具体合约地址
- `sponsorGas`：是否启用 gas 赞助

### 步骤 2：连接 Smart Account

通过 `smartWallet()` 包装个人账户，然后连接：

```ts
const smart = smartWallet({
  chain: sepolia,
  sessionKey: {
    address: sessionKeyAccountAddress,
    permissions: {
      approvedTargets: "*",
    },
  },
  sponsorGas: true,
});

const smartAccount = await smart.connect({
  client,
  personalAccount,
});
```

这里可以理解为：

> 用户的 personal account 是主控身份，smart account 是真正执行链上操作的智能账户。

### 步骤 3：验证 Session Key 是否注册成功

可以通过 `getAllActiveSigners()` 查看当前智能账户的 active signers。

核心检查：

```ts
const isSessionKeyActive = signers
  .map((s) => s.signer)
  .includes(sessionKeyAccountAddress);
```

如果 Session Key 不在 active signers 里，后续执行会失败。

### 步骤 4：创建 Engine Server Wallet

thirdweb Engine 允许后端用 Session Key 执行交易。

ERC-4337 模式下需要指定：

```ts
executionOptions: {
  entrypointVersion: "0.6",
  signerAddress: sessionKeyAccountAddress,
  smartAccountAddress: smartAccount.address,
  type: "ERC4337",
}
```

这里几个字段很重要：

- `entrypointVersion`：使用的 EntryPoint 版本
- `signerAddress`：Session Key 地址
- `smartAccountAddress`：被代表执行操作的智能账户地址
- `type`：账户抽象类型，这里是 `ERC4337`

### 步骤 5：用 Session Key 执行交易

创建好 server wallet 之后，就可以：

```ts
const tx = await sendTransaction({
  account: serverWallet,
  transaction: {
    chain: sepolia,
    client,
    to: targetAddress,
    value: 0n,
  },
});
```

从产品角度看，这意味着：

> 用户一次授权后，后端可以在权限范围内代用户执行交易，不需要用户每次打开钱包签名。

## 9. thirdweb Session Keys Guide：ERC-7702 Smart EOA 版本

thirdweb 文档还讲了 ERC-7702 / EIP-7702 Smart EOA 的模式。

简单理解：

> ERC-7702 让 EOA 可以临时具备智能账户能力，从而实现类似智能钱包的功能。

thirdweb 里可以用 In-App Wallet 配置：

```ts
const wallet = inAppWallet({
  executionMode: {
    mode: "EIP7702",
    sponsorGas: true,
  },
});
```

然后连接用户：

```ts
const account = await wallet.connect({
  chain: sepolia,
  client,
  strategy: "google",
});
```

这类方式更接近 Web2 登录体验，比如：

- Google 登录
- Email 登录
- Guest 登录

### ERC-7702 Session Key 创建流程

大致流程：

1. 创建 ERC-7702 Smart EOA
2. 发一笔交易触发账户部署或升级
3. 获取账户合约实例
4. 调用 `createSessionKey()`
5. 配置有效期、权限策略
6. 用 Engine 的 server wallet 执行交易

### Full Permission 模式

可以直接给 Session Key 完整权限：

```ts
createSessionKey({
  account,
  contract: accountContract,
  sessionKeyAddress,
  durationInSeconds: 86400,
  grantFullPermissions: true,
})
```

含义：

- Session Key 有效期 1 天
- 在有效期内拥有比较大的执行权限

适合：

- demo
- 测试网
- 内部开发

不适合：

- 生产环境
- 涉及真实资产的账户

### Granular Permission 模式

生产环境更应该用细粒度权限：

```ts
grantFullPermissions: false,
callPolicies: [...],
transferPolicies: [...]
```

可以限制：

- 只能调用某个合约
- 只能调用某个函数 selector
- 每次最大转账金额
- 生命周期内最大额度
- 每天或每周期 allowance
- 参数必须满足条件

## 10. Session Key 权限设计

thirdweb 文档里提到几类关键权限。

### 10.1 Approved Targets

控制 Session Key 可以调用哪些合约。

示例：

```ts
approvedTargets: "*"
```

或者：

```ts
approvedTargets: ["0x123...", "0x456..."]
```

学习理解：

- `"*"` 方便，但风险大
- 具体地址更安全
- 如果是链游或高频应用，最好只允许特定合约

### 10.2 Call Policies

控制 Session Key 可以调用哪些智能合约函数。

常见字段：

- `target`：目标合约地址
- `selector`：函数 selector，比如 ERC20 `transfer` 是 `0xa9059cbb`
- `maxValuePerUse`：单次调用最多能携带多少原生币
- `valueLimit`：总额度限制
- `constraints`：参数级别限制

这非常适合做安全策略。

比如：

> 允许 Session Key 调用某个 ERC20 的 transfer，但只能转给固定地址，且总额度不超过某个值。

### 10.3 Transfer Policies

控制原生 token 转账权限。

可以限制：

- 收款地址
- 单次最大金额
- 总额度
- allowance 重置周期

### 10.4 LimitType

thirdweb 提到三类额度限制：

```ts
enum LimitType {
  Unlimited = 0,
  Lifetime = 1,
  Allowance = 2,
}
```

理解：

- `Unlimited`：不限制，不推荐生产环境随便用
- `Lifetime`：生命周期内总额度，用完就没了
- `Allowance`：周期性额度，比如每天重置一次

### 10.5 Constraint Conditions

参数约束条件包括：

```ts
enum Condition {
  Unconstrained = 0,
  Equal = 1,
  Greater = 2,
  Less = 3,
  GreaterOrEqual = 4,
  LessOrEqual = 5,
  NotEqual = 6,
}
```

这类参数约束很重要，因为它可以把 Session Key 从“能调用函数”进一步限制到“只能用特定参数调用函数”。

例如：

- 只能转给某个地址
- 只能 swap 某个 token pair
- 只能 mint 某个 NFT
- 只能调用小于某额度的操作

## 11. 安全注意事项

Session Key 本质上是授权能力，所以安全设计非常关键。

### 11.1 不要随便给全权限

`grantFullPermissions: true` 或 `approvedTargets: "*"` 很方便，但风险也最大。

更好的做法：

- 限制目标合约
- 限制函数 selector
- 限制金额
- 限制有效期
- 限制调用参数
- 限制调用频率

### 11.2 设置合理有效期

比如：

- 临时会话：几小时
- 当天操作：1 天
- 长期设备：更长，但需要更严格权限和监控

不要让 Session Key 永久有效，除非有非常明确的 revoke 和监控机制。

### 11.3 Session Key 要安全存储

thirdweb 文档建议使用 vault 系统。

原因是：

> Session Key 虽然不是主钱包私钥，但它仍然能代表账户做事，泄露后也可能造成损失。

### 11.4 定期轮换和撤销

建议：

- 定期 rotate keys
- 不用的 Session Key 及时 revoke
- 发现异常立即 revoke
- 对高价值账户避免长时间授权

### 11.5 监控 Session Key 使用情况

应该监控：

- 调用频率是否异常
- 调用目标是否异常
- 金额是否异常
- 是否触发权限拒绝
- 是否出现连续失败交易

## 12. 常见错误与排查

### 12.1 Session key not active

可能原因：

- Session Key 没有成功注册到智能账户
- 用户没有完成授权
- active signers 查询不到该地址

排查：

- 用 `getAllActiveSigners()` 检查
- 确认 Session Key 地址是否一致
- 确认连接的是正确的 smart account

### 12.2 Permission denied

可能原因：

- 目标合约不在 `approvedTargets`
- call policy 不允许该 selector
- transfer policy 不允许该接收地址
- 参数 constraints 不满足

排查：

- 检查 target 地址
- 检查 function selector
- 检查 value limit
- 检查参数约束

### 12.3 Gas estimation failed

可能原因：

- Paymaster 或 gas sponsorship 配置错误
- 账户没有足够余额
- Bundler 模拟失败
- 目标调用本身会 revert

排查：

- 先不用 Paymaster 测试
- 检查 sponsorGas 配置
- 检查链、EntryPoint 版本
- 查看交易模拟报错

### 12.4 Vault token invalid

可能原因：

- Engine 的 Vault token 不正确
- token 权限不足
- 环境变量没读取到

排查：

- 检查 `process.env.VAULT_TOKEN`
- 确认 token 权限
- 不要把 secretKey / vault token 放到前端

### 12.5 ERC-7702 account not deployed

可能原因：

- 还没有触发账户部署或升级
- 没有先发 self-transfer 或初始化交易

排查：

- 先执行一笔部署或升级交易
- 再创建 account contract
- 再创建 Session Key

## 13. ERC-4337 与 ERC-7702 的区别理解

### ERC-4337 Smart Account

核心是：

> 用户使用智能合约钱包作为账户。

特点：

- 账户本身是合约
- 通过 EntryPoint 执行 UserOperation
- 适合成熟的智能钱包架构
- 支持 Paymaster、Bundler、Session Key、模块化等

### ERC-7702 Smart EOA

核心是：

> 让 EOA 临时或可控地拥有智能合约账户能力。

特点：

- 对传统 EOA 更兼容
- 可以让 EOA 获得智能钱包能力
- 更适合从现有钱包体系平滑过渡
- 可以结合 Google、email 等登录方式使用

## 14. 我的理解总结

这两份资料放在一起看，重点不是单独学某个 API，而是理解一个新的钱包范式。

### 旧模式

```text
用户 EOA 私钥
  ↓
每次操作都手动签名
  ↓
必须自己付 gas
  ↓
钱包功能弱，UX 差
```

### ERC-4337 模式

```text
用户智能账户
  ↓
UserOperation 表达用户意图
  ↓
Bundler 打包
  ↓
EntryPoint 验证与执行
  ↓
Paymaster 可代付 gas
  ↓
账户逻辑可编程
```

### Session Key 模式

```text
主账户授权 Session Key
  ↓
Session Key 在有限权限内执行操作
  ↓
用户不必每次手动签名
  ↓
适合链游、移动端、高频交互
```

## 15. 和 Monad Builder Camp 的关系

这部分内容虽然来自 Ethereum / thirdweb 文档，但对 Monad Builder Camp 的学习也很有帮助。

我今天的理解是：高性能链不只是 TPS 更高、确认更快，也需要更好的账户和交互体验。如果底层执行很快，但用户仍然要频繁处理 gas、签名、nonce、钱包弹窗，普通用户进入链上应用的门槛还是很高。

ERC-4337、Paymaster、Session Key 这类账户抽象能力，可以帮助开发者设计更顺滑的链上应用：

- 新用户可以更容易完成首次链上操作
- 应用可以通过 Paymaster 降低 gas 门槛
- 高频交互可以通过 Session Key 减少重复签名
- 账户权限可以更加细粒度，而不是只有“主私钥完全控制”一种模式

对于 Monad 生态里的应用开发来说，这提醒我除了关注合约部署、交易速度和链上交互，也要关注钱包层、账户层和用户操作路径。一个好的 Web3 应用不只是能上链，还应该让用户更容易理解和使用。

## 16. 关键词速记

- **Account Abstraction / AA**：账户抽象，让钱包变成可编程账户
- **ERC-4337**：不改以太坊协议实现账户抽象的标准
- **Smart Account**：智能合约钱包账户
- **UserOperation**：用户意图对象，不是普通交易
- **Alt Mempool / UserOp Mempool**：专门传播 UserOperation 的 mempool
- **Bundler**：收集、模拟、打包 UserOperation 的节点
- **EntryPoint**：统一验证和执行 UserOperation 的入口合约
- **Paymaster**：在特定条件下帮用户付 gas 的合约
- **Session Key**：临时或受限授权密钥
- **Delegation**：授权其他 key 或模块代表账户执行操作
- **ERC-7702 / EIP-7702**：让 EOA 获得类似智能账户能力的机制

---

## 17. 一句话总结

> ERC-4337 解决的是“钱包如何变成可编程智能账户”；Session Key 解决的是“如何在不暴露主密钥、不频繁打扰用户的情况下，安全地授权应用执行操作”。
