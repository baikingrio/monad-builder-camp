## 我对“真实上链前端”的理解

之前的页面能够连接钱包、展示实验 proof，也能逐步点击模拟流程，但本质还是本地状态变化，并不会真的生成或提交 UserOperation。

如果要让用户在页面里真正体验 ERC-4337，前端不能直接保存 sponsor 私钥，也不能让一个公开 Paymaster 无限制地替任何人付 gas。更合理的分层是：

```text
浏览器钱包
  ↓ 用户确认、签名 UserOperation
Nuxt 4 前端
  ↓ 提交受限请求
Nuxt Nitro / Relay 服务
  ↓ 校验规则、签发短时赞助授权、发送 handleOps
Monad EntryPoint v0.8
  ↓
智能账户 / Paymaster / Session Key 目标合约
```

这样用户始终掌握自己的 EOA 签名；后端只负责受限的赞助和 relay；Paymaster 的资金也不会因为一个开放接口被任意消耗。

## 一、真实执行层的合约设计

### 1. CampaignSponsoredPaymaster

我新建了一个学习版 `CampaignSponsoredPaymaster`。它和之前“只赞助一个固定账户”的最小 Paymaster 不同，改成由 sponsor 服务为每一笔符合规则的 UserOperation 签发短时授权。

授权绑定以下内容：

- `sender` 智能账户地址
- EntryPoint nonce
- `initCode` 的哈希
- `callData` 的哈希
- 有效期 `validUntil`
- 当前链 ID
- 当前 Paymaster 合约地址

这意味着，即使有人拿到一份授权，也不能把它换成其他账户、其他 calldata 或其他链上的调用。

Paymaster 的 `paymasterAndData` 使用下面的结构：

```text
paymaster 地址
+ verificationGasLimit
+ postOpGasLimit
+ validUntil
+ sponsor signature
```

合约只接受 EntryPoint 调用；过期授权、无效签名、被替换的 `callData` 都会被拒绝。

### 2. SessionKeyAccountFactory

Session Key 不适合复用课程账户之前部署的单一实验账户，因为每位用户都应该拥有自己的 owner、session key、有效期和权限范围。

因此今天增加了 `SessionKeyAccountFactory`：

- 用 CREATE2 为每个用户预测 Session Key 智能账户地址。
- 根据 `owner`、`sessionKey`、允许目标合约、允许函数 selector、有效期、额度和 salt 计算地址。
- 同一组参数重复调用时，返回已经部署的账户，不重复部署。

这样后续前端可以在用户授权前先展示地址，在用户首次使用时再部署对应账户。

### 3. Session Key 的真实限制

Session Key 账户仍然保留 owner 的完整权限，但临时 key 只允许调用：

```text
SessionKeyDemoTarget.checkIn(string)
```

并且限制：

- 目标合约必须匹配白名单。
- 函数 selector 必须匹配。
- 必须未过期。
- `nativeSpendLimit = 0`，不能用 session key 转 MON。

这比“前端保管一个能做任何事的私钥”安全得多，也更符合 Session Key 的实际用途：降低每日签到、游戏动作、积分任务等低风险操作的重复签名成本。

## 今日收获

今天让我更清楚地理解到，ERC-4337 的难点不只是构造一个 `PackedUserOperation`，而是要同时处理好三个边界：

1. **用户授权边界**：用户需要对自己的操作签名，Paymaster 不能替用户授权。
2. **资金边界**：Paymaster 可以赞助 gas，但必须通过短时授权、动作白名单、额度和频率限制防止预算被滥用。
3. **权限边界**：Session Key 可以让低风险操作更顺滑，但必须严格限制目标、函数、有效期和花费额度。

Nuxt 4 的价值在于把浏览器交互和服务端 Relay 放在一个项目里：前端负责让用户看懂、签名和查看交易；服务端负责保管 sponsor 私钥、校验策略和转发操作。这样比把私钥或无限制 Paymaster 逻辑放到前端更符合真实应用的安全结构。

## 四、ERC-1363 质押收益分红实践

今天还阅读了 [EIP-1363](https://eips.ethereum.org/EIPS/eip-1363)，并用它实现了一个测试网质押收益分红场景。

ERC-1363 可以理解成在 ERC-20 的基础上增加“转账后立即通知接收合约”的能力。核心接口包括：

- `transferAndCall`
- `transferFromAndCall`
- `approveAndCall`

其中 `transferAndCall` 会先完成 Token 转账，然后回调接收方的：

```text
onTransferReceived(operator, from, value, data)
```

接收方必须返回正确的 magic value `0x88a7ca5c`，否则整笔交易会回滚。这个特性很适合质押：用户不需要先执行 `approve`，再执行 `stake`，而是在一笔交易内完成“转入代币 + Vault 记录质押”。

### 1. 合约设计

我新增了两个合约：

- `StakeRewardToken`：学习版 ERC-20 + ERC-165 + ERC-1363 Token。
- `ERC1363StakingDividendVault`：接收 ERC-1363 回调、记录质押并分配收益的 Vault。

Vault 的规则：

1. 用户调用：

```text
transferAndCall(vault, amount, 0x)
```

空 data 代表质押。Token 转入 Vault 后，Vault 的 `onTransferReceived` 会将这笔金额计入用户的 `stakedBalance`。

2. Vault owner 调用：

```text
transferAndCall(vault, rewardAmount, abi.encode(uint8(1)))
```

这代表注入分红。Vault 按当前 `totalStaked` 更新 `accRewardPerShare`，让收益按质押份额分配。

3. 用户调用 `claim()` 领取收益；调用 `withdraw(amount)` 提取本金。提取前会先结算收益，因此不会丢失已经累计但尚未领取的分红。

我也加入了几个必要限制：只接受指定 Token、没有任何质押时不允许注入收益、只有 owner 能注入收益、错误回调会回滚，并用 reentrancy lock 保护回调、领取和提取过程。

## 今天新增的收获

今天把两个看起来不同的标准放在一起后，我更容易理解它们的产品价值：

- ERC-4337 解决的是账户、Gas 和权限如何更像普通 App。
- ERC-1363 解决的是代币支付后如何在同一笔交易里立刻触发业务逻辑。

对于质押产品，ERC-1363 能减少一步 approve + stake 的交互；对于链游、积分、会员、订阅等场景，它也可以把“支付代币”和“开通服务/记录权益”放进同一笔原子交易。

但这类 callback 机制也要求接收合约认真校验 Token 地址、data 格式和回调返回值；否则很容易出现错误 Token 进入系统、业务含义不清或资金状态与业务状态不一致的问题。
