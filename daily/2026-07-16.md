## 今天重点理解：Moss 的 Adapter 架构

我进一步阅读了 Moss 的 Protocol Onboarding、MCP Tools、Agent Safety Rules、Protocol Template，以及 Kuru 的真实 Adapter 源码。

我现在对 Adapter 的理解是：在 Moss 中它更准确叫 **Protocol package**。它不是简单地把某个 Solidity 函数包起来，而是负责把一个链上协议整理成 Agent 能够安全使用和核对的能力层。

一个 Adapter / Protocol package 需要负责：

- ABI 与合约地址的来源、验证与维护；
- Protocol 的自我描述与依赖关系；
- Capability，例如 `swap`、`approve`、`transfer`；
- 参数类型、单位、默认值和风险标签；
- 多笔操作组成的 Capability Tree；
- 模拟后对 Event、原生币转账等 Change 的 Receipt 解释；
- 覆盖失败、顺序异常、解析错误等安全测试。

以 Kuru swap 为例，用户看见的是“用某个 Token 换另一个 Token”；Adapter 则负责在底层处理市场路径、Token 精度、滑点、必要时的 ERC-20 授权、Router 调用和模拟结果解析。

## 为什么 Receipt Parser 很重要

一次交易成功，并不表示它一定符合用户意图。

Moss 的 Simulator 会从真实 trace 中提取 Event 和原生 MON 转账等 Change；Adapter 的 Receipt Parser 再把这些底层变化翻译成可读结果，例如实际转出了什么、收到了什么、是否发生了授权、成交事件是否符合预期。

更关键的是，Moss 不允许 Receipt 只挑选“看起来重要”的部分。它要求对原始 Change 做：

```text
完整覆盖 + 不重复 + 不替换 + 顺序一致
```

如果交易回滚、无法得到 trace、无法证明顺序、Receipt 解析失败，或 Change 被遗漏、重复和乱序，Moss 就会产生 Warning 并停止。这种设计让我印象很深：**看不清楚，就不要假装已经理解。**

## 今日学习总结

今天最大的收获是：Moss 的价值不只是让 Agent 更方便地调用协议，而是把“协议怎么调用”和“执行结果怎么证明”都变成可检查的结构。

一个好的 Adapter 不应只回答“怎样发起交易”，还要回答：合约和 ABI 从哪里来、参数是什么含义、会有什么风险、是否需要授权、模拟后每一笔资产变化如何解释。即使 `warningCount = 0`，用户仍必须核对实际资产、金额、接收方、授权对象和协议选择，并由钱包保留最终签名权。
