## 我对 counterfactual deployment 的理解

普通合约账户一般是先部署，部署交易确认后才知道地址。虽然合约地址也可以根据 deployer 和 nonce 计算，但从产品体验上看，用户通常会先经历“创建钱包/部署钱包”的步骤。

counterfactual deployment 的重点是：

> 智能账户还没有真正部署，但我们已经可以提前算出它未来会部署到哪个地址。

在这次实验里，我用了一个很小的 factory：`Minimal4337AccountFactory`。它用 `CREATE2` 部署 `Minimal4337Account`，所以只要这些参数确定，账户地址就确定：

- factory 地址
- salt
- `Minimal4337Account` creation code
- constructor 参数：`entryPoint` 和 `owner`

这意味着：链上还没有账户代码时，前端或后端也可以先把这个地址当成用户的智能账户地址。

## 一个简单应用场景

我用“链上活动 App / 新用户领取徽章”来理解这个功能。

假设一个新用户第一次打开活动页面，还没有部署 AA 钱包。App 可以先根据用户 EOA、EntryPoint 和 salt 算出他的智能账户地址，然后：

1. 页面先显示这个未来的智能账户地址。
2. 后端、空投系统或朋友可以先往这个地址转测试币/活动资产。
3. 这个时候账户合约还没部署，所以 `code.length = 0`。
4. 用户第一次点击“领取徽章 / 签到 / 转账”时，UserOperation 带上 `initCode`。
5. EntryPoint 在同一笔 `handleOps` 里先通过 factory 部署账户，再执行这次操作。

这样用户不需要一开始就单独发一笔“部署钱包”的交易。对产品来说，可以做成更接近普通 App 的体验：先给用户一个可收款地址，真正需要链上操作时再部署。

## initCode 的作用

这次实验里，`initCode` 可以理解为：

```text
factory 地址 + factory.createAccount(entryPoint, owner, salt) 的 calldata
```

也就是说，`initCode` 告诉 EntryPoint / Bundler：

> 如果 `sender` 这个智能账户还没有部署，就先调用这个 factory，用这些参数把账户创建出来。

本次 UserOperation 的关键点：

- `sender` 填的是提前算出来的 counterfactual account 地址。
- `initCode` 填的是“如何部署这个账户”。
- `callData` 填的是账户部署后要执行的操作。

所以第一次 `handleOps` 实际上完成了两个动作：

1. 部署智能账户。
2. 让这个智能账户执行一次 `execute`。

## 本次链上实验流程

1. 部署 `Minimal4337AccountFactory`。
2. 用 `factory.getAddress(entryPoint, owner, salt)` 预测智能账户地址。
3. 确认这个地址部署前 `code.length = 0`。
4. 先向这个尚未部署的地址转入 `0.2 MON`。
5. 构造带 `initCode` 的 UserOperation。
6. 通过 EntryPoint v0.8 调用 `handleOps`。
7. 验证账户部署成功、nonce 增加、账户余额仍然存在。

## 今天的收获

这一步让我更清楚地理解了 AA 钱包的一个核心体验优势：

> 用户的智能账户地址可以先被使用，账户合约可以等到第一次真正交互时再部署。

如果把它放到产品里，就可以减少“先部署钱包再使用”的割裂感。比如活动 App、游戏、积分系统、任务平台，都可以先给用户一个确定的智能账户地址，等用户第一次操作时再用 `initCode` 完成部署和执行。

这也是 ERC-4337 和普通 EOA 体验差异很大的地方：账户本身也可以变成应用逻辑的一部分，而不只是一个静态地址。
