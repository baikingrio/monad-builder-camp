## 今日学习主题

今天围绕 ERC-4337 Account Abstraction 做了三组连续实验，把智能账户体验从“能执行 UserOperation”继续推进到更接近真实应用的能力：

1. **counterfactual deployment / initCode**：理解“账户还没部署，但地址已经可以提前使用”。
2. **Paymaster / sponsored UserOp**：理解“应用方可以在规则内替用户支付 gas”。
3. **Session Key / 受限权限**：理解“用户可以授权临时低权限密钥，减少高频低风险操作的重复签名”。

前两步已经完成过：

1. 在 Monad Testnet 上跑通最小 ERC-4337 账户和 `handleOps`。
2. 把脚本迁移到 Monad Testnet 官方 EntryPoint v0.8。

今天的重点不是只跑通代码，而是把每个 AA 功能都放进一个具体应用场景里理解：链上活动、签到、链游、积分任务、新用户 onboarding 等。

## 今日总结

今天最大的收获是：ERC-4337 不只是把 EOA 换成智能合约账户，而是把“账户创建、gas 支付、操作权限”都变成可以编程的产品能力。

- `initCode` 解决的是 **账户创建体验**：用户可以先拥有一个可预测的智能账户地址，真正第一次交互时再部署账户。
- `Paymaster` 解决的是 **gas 体验**：用户不一定要先准备 gas token，应用可以在白名单、限额、签名等规则内赞助关键操作。
- `Session Key` 解决的是 **频繁签名体验**：用户不需要每个低风险动作都弹钱包，但也不会把完整 owner 权限交给前端。

这三个能力组合起来，很适合 Monad 这类高性能 EVM 链上的高频应用：链游、任务平台、活动签到、积分系统、社交应用、轻量 DeFi 自动化等。Monad 提供更高吞吐和更低延迟，而 AA 提供更接近普通 App 的账户与交互体验。

今天也验证了一个很重要的安全边界：体验优化不能等于无限授权。无论是 Paymaster 还是 Session Key，生产环境都必须有明确限制：目标合约白名单、函数白名单、额度、有效期、防重放、后端签名、可撤销机制和监控。

## 今日完成的链上 Proof

- Counterfactual / initCode：预测地址、预充值、首次 UserOperation 部署账户并执行成功。
- Paymaster / sponsored UserOp：Paymaster deposit 充值成功，UserOperation gas 由 Paymaster 赞助，nonce 增加。

## 我对 counterfactual deployment 的理解

普通合约账户一般是先部署，部署交易确认后才知道地址。虽然合约地址也可以根据 deployer 和 nonce 计算，但从产品体验上看，用户通常会先经历“创建钱包/部署钱包”的步骤。

counterfactual deployment 的重点是：

> 智能账户还没有真正部署，但我们已经可以提前算出它未来会部署到哪个地址。

在这次实验里，我用了一个很小的 factory：`Minimal4337AccountFactory`。它用 `CREATE2` 部署 `Minimal4337Account`，所以只要这些参数确定，账户地址就确定：

- factory 地址
- salt
- `Minimal4337Account` creation code
- constructor 参数：`entryPoint` 和 `owner`

这意味着：链上还没有账户代码时，前端或后端也可以先把这个地址当成用户的智能账户地址。

## 一个简单应用场景

我用“链上活动 App / 新用户领取徽章”来理解这个功能。

假设一个新用户第一次打开活动页面，还没有部署 AA 钱包。App 可以先根据用户 EOA、EntryPoint 和 salt 算出他的智能账户地址，然后：

1. 页面先显示这个未来的智能账户地址。
2. 后端、空投系统或朋友可以先往这个地址转测试币/活动资产。
3. 这个时候账户合约还没部署，所以 `code.length = 0`。
4. 用户第一次点击“领取徽章 / 签到 / 转账”时，UserOperation 带上 `initCode`。
5. EntryPoint 在同一笔 `handleOps` 里先通过 factory 部署账户，再执行这次操作。

这样用户不需要一开始就单独发一笔“部署钱包”的交易。对产品来说，可以做成更接近普通 App 的体验：先给用户一个可收款地址，真正需要链上操作时再部署。

## initCode 的作用

这次实验里，`initCode` 可以理解为：

```text
factory 地址 + factory.createAccount(entryPoint, owner, salt) 的 calldata
```

也就是说，`initCode` 告诉 EntryPoint / Bundler：

> 如果 `sender` 这个智能账户还没有部署，就先调用这个 factory，用这些参数把账户创建出来。

本次 UserOperation 的关键点：

- `sender` 填的是提前算出来的 counterfactual account 地址。
- `initCode` 填的是“如何部署这个账户”。
- `callData` 填的是账户部署后要执行的操作。

所以第一次 `handleOps` 实际上完成了两个动作：

1. 部署智能账户。
2. 让这个智能账户执行一次 `execute`。

## 本次链上实验流程

1. 部署 `Minimal4337AccountFactory`。

5. 构造带 `initCode` 的 UserOperation。
6. 通过 EntryPoint v0.8 调用 `handleOps`。
7. 验证账户部署成功、nonce 增加、账户余额仍然存在。

# 第四步补充｜paymaster / sponsored UserOp 概念验证

## 为什么继续做 Paymaster

完成 `initCode` 后，我接着补 ERC-4337 的第四个重要概念：**Paymaster / sponsored UserOp**。

前三步让我理解了：

1. 智能账户如何通过 EntryPoint 执行 UserOperation。
2. v0.8 EntryPoint 的基本调用方式。
3. 账户如何用 `initCode` 在第一次 UserOperation 中懒部署。

但如果用户第一次使用应用时还要自己准备 MON 付 gas，体验仍然不够接近普通 App。所以这一步我想验证：

> 能不能让项目方先准备一笔 gas 预算，由 Paymaster 替用户的 UserOperation 支付 gas？

## 我对 Paymaster 的理解

Paymaster 可以理解成 ERC-4337 里的“gas 赞助方”。

普通 UserOperation 的 gas 最终由智能账户承担；而带 Paymaster 的 sponsored UserOp，会让 EntryPoint 在验证时额外调用 Paymaster：

```text
validatePaymasterUserOp(userOp, userOpHash, maxCost)
```

Paymaster 如果返回 `validationData = 0`，就表示它愿意赞助这次 UserOp。之后这次 UserOp 的 gas 会从 Paymaster 在 EntryPoint 里的 deposit 扣，而不是从用户智能账户扣。

这里有一个很重要的点：

> Paymaster 不替用户授权操作。用户仍然要用自己的 owner key 签名 UserOperation。Paymaster 只决定“我愿不愿意替这次操作付 gas”。

这能避免把“授权”和“付费”混在一起。

## 一个简单应用场景

我用“链上活动 App 赞助新用户第一次签到”来理解这个功能。

假设一个用户第一次进入活动页面，App 希望他可以直接点击“签到 / 领取徽章”，而不是先去交易所或水龙头准备 MON。

产品流程可以是：

1. 活动方部署一个 Paymaster。
2. 活动方往 EntryPoint 的 Paymaster deposit 里充值一笔 MON，作为活动 gas 预算。
3. 用户点击“签到”。
4. 前端构造 UserOperation，并在 `paymasterAndData` 里带上 Paymaster 地址和活动信息。
5. 用户仍然签名确认这次操作。
6. EntryPoint 调用智能账户验签，再调用 Paymaster 判断是否赞助。
7. 如果 Paymaster 认可这次操作，用户不需要自己付这次 UserOp 的 gas。

这类能力可以用于：

- 新用户 onboarding
- 免费领取徽章 / NFT
- 活动签到
- 游戏新手任务
- 积分任务平台
- 限额补贴的链上操作

## paymasterAndData 的作用

这次实验里，`paymasterAndData` 的结构是：

```text
paymaster 地址 + paymasterVerificationGasLimit + paymasterPostOpGasLimit + 自定义业务数据
```

在脚本中我拼了一个简单的业务标识：

```text
campaign-checkin-sponsored-demo
```

真实产品中，这个位置通常不会只放字符串，而可能放：

- 活动 ID
- 用户额度凭证
- 后端签名
- 有效期
- 防重放 nonce
- 白名单证明

这让我理解到：`paymasterAndData` 不只是“填一个 Paymaster 地址”，它更像 Paymaster 的业务验证入口。

## 本次链上实验流程

1. 部署一个新的 `Minimal4337Account`，账户里只放 `0.05 MON`。
2. 部署 `SimpleSponsoredPaymaster`，设置只赞助这个账户。
3. 调用 `depositToEntryPoint`，给 Paymaster 在 EntryPoint 中充值 `0.3 MON`。
4. 构造带 `paymasterAndData` 的 UserOperation。
5. 用户用 owner key 签名 UserOperation。
6. EntryPoint 调用账户 `validateUserOp` 验签。
7. EntryPoint 调用 Paymaster `validatePaymasterUserOp`，Paymaster 确认 sender 是被赞助账户。
8. 执行账户 `execute`，从账户转 `0.001 MON` 给 owner。
9. 验证账户 nonce 增加，账户余额只少了业务转账，Paymaster deposit 承担 gas 消耗。

# 第五步补充｜session key / 受限权限实验

## 为什么继续做 Session Key

完成 Paymaster 后，我继续补 ERC-4337 里另一个很重要的体验能力：**session key / 受限权限**。

前面几步分别解决了：

1. 智能账户如何通过 EntryPoint 执行 UserOperation。
2. 如何用 `initCode` 做懒部署。
3. 如何用 Paymaster 赞助 gas。

但真实产品里还有一个体验问题：如果用户每次签到、游戏动作、积分任务都要弹钱包签名，体验仍然会很割裂。

Session key 想解决的是：

> 用户先给应用一个临时、受限的密钥。之后应用可以用这个密钥签低风险操作，但这个密钥不能拿走用户账户的完整控制权。

## 我对 Session Key 的理解

Session key 不是新的 owner，也不是把账户交给应用控制。

它更像一个“临时通行证”：

- 有效期有限。
- 能调用的目标合约有限。
- 能调用的函数有限。
- 能花的钱有限。
- 可以被撤销或过期。

这次实验里，我做了一个 `MinimalSessionKeyAccount`，它保留 owner 全权限，同时额外允许一个 session key 签名 UserOperation。但 session key 必须满足这些限制：

- 只能调用指定的 `SessionKeyDemoTarget`。
- 只能调用 `checkIn(string)`。
- 有效期是部署后一段时间。
- `nativeSpendLimit = 0`，也就是不能用它转 MON。

验证逻辑放在账户的 `validateUserOp` 中。账户会解析 UserOperation 的 `callData`，确认它是不是：

```text
account.execute(allowedTarget, 0, checkIn("session-key-demo"))
```

如果目标、函数、有效期或额度不符合，session key 签名就会被拒绝。

## 一个简单应用场景

我用“链游 / 活动 App 的每日签到”来理解 session key。

假设用户进入一个链上小游戏，里面有很多低风险动作：

- 每日签到
- 移动一步
- 领取小积分
- 完成一个普通任务
- 记录一次游戏动作

如果每一步都弹钱包签名，用户体验会很差。更自然的方式是：

1. 用户先用 owner 钱包授权一个 session key。
2. session key 只允许调用游戏合约里的低风险函数。
3. session key 一天后过期。
4. session key 不能转账，不能换 owner，不能调用其他合约。
5. 用户玩游戏时，前端用 session key 帮用户签这些低风险 UserOperation。

这样既减少了频繁签名，又不会把完整账户权限交给前端。

## 本次链上实验流程

1. 部署 `SessionKeyDemoTarget`，模拟活动签到 / 游戏每日动作。
2. 部署 `MinimalSessionKeyAccount`。
3. 设置：
   - owner：课程账户
   - sessionKey：实验用临时地址
   - allowedTarget：`SessionKeyDemoTarget`
   - allowedSelector：`checkIn(string)`
   - nativeSpendLimit：`0`
4. 构造 UserOperation：`execute(target, 0, checkIn("session-key-demo"))`。
5. 用 session key 签名，而不是 owner key。
6. EntryPoint 调用账户 `validateUserOp`。
7. 账户验证 session key 签名和限制规则。
8. 验证通过后执行 `checkIn`。
9. 链上确认 `checkInCount = 1`，EntryPoint nonce 增加到 `1`。

## 这一步的重要收获

Session key 让我更清楚地看到 AA 钱包和普通 EOA 的差异：

> 普通 EOA 基本是“一个私钥 = 全部权限”，而智能账户可以把权限拆成很多细粒度规则。

这对产品体验很重要。比如链游、社交、积分、任务系统里，有很多操作并不值得每次都弹钱包签名。Session key 可以让这些低风险操作更顺滑。

但 session key 也很容易被误用。生产环境必须认真设计限制条件：

- 有效期必须短。
- 目标合约要白名单。
- 函数 selector 要白名单。
- 花费额度要有限。
- 最好可以撤销。
- 前端或设备丢失时要能快速失效。
- 高风险操作仍然应该要求 owner 重新签名。

这次实验只做了最小概念验证：一个 session key 只能调用 `SessionKeyDemoTarget.checkIn(string)`，不能转 MON，不能调用其他目标或函数。它足够用来理解 session key 的核心思想：**减少频繁签名，但不交出完整权限**。
