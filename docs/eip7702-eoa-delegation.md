# ERC-7702 学习笔记：让 EOA 获得可编程执行能力

> 研究日期：2026-07-12
> 定位：概念、安全边界与 Monad Testnet 真实实践记录。以下 Proof 已在 `2026-07-12` 验证，不包含私钥或 relayer 密钥。

## 先说结论

ERC-7702 让一个传统钱包地址（EOA）可以授权自己的地址“委托”给某个合约实现执行。这样用户仍用原来的 EOA 私钥签名，但该地址在执行时可以拥有合约钱包的能力，例如批量操作、受限权限和更灵活的 Gas 体验。

它不是“把 EOA 变成一个新地址”，也不是自动提供免 Gas。更准确地说：**EOA 地址不变，私钥不变；账户代码槽写入一个指向委托实现的标记，后续执行会加载该实现的逻辑。**

ERC-7702 已随 Ethereum Pectra 升级在以太坊主网上于 2025-05-07 激活。是否能在某条 EVM 链使用，仍要单独确认该链已经支持该交易类型及钱包 / RPC 工具链。

## 它想解决什么问题

传统 EOA 的逻辑固定：一把私钥直接控制账户，常见操作往往要拆成多笔交易。智能合约账户功能更强，但迁移、部署和基础设施接入又会增加复杂度。

ERC-7702 的目标是在两者之间提供一条渐进路线。规范重点提到三类能力：

1. **批量执行（batching）**：把 `approve` 和后续兑换等多个操作放到一笔原子交易中；任一步失败则整体回滚。
2. **Gas 赞助（sponsorship）**：应用或第三方可替用户承担交易 Gas，但仍必须有独立的赞助策略、风控和资金来源。
3. **权限降级（privilege de-escalation）**：为子密钥或会话密钥设置更小的权限，例如只允许与某个应用交互、只允许花特定 ERC-20、或限定每日额度。

## 协议层发生了什么

ERC-7702 定义了新的 `Set Code Transaction`：

- 交易类型：`0x04`
- EOA 对授权内容签名，授权元组形如：

  ```text
  [chain_id, address, nonce, y_parity, r, s]
  ```

- 授权内容包含链 ID、委托实现地址和 nonce；这些字段用于限制链范围并防止授权被重放。
- 在交易中处理授权后，EOA 的代码会被写成 delegation indicator：

  ```text
  0xef0100 || delegate_address
  ```

这不是把完整合约字节码复制到 EOA，而是写入一个很短的“去某个实现地址取代码执行”的指针。执行逻辑仍在 EOA 的地址、余额和存储上下文中生效，因此委托实现必须被视为该 EOA 的高权限钱包逻辑。

授权可以由 relayer 放进交易中提交，这有助于改善 UX；但 relayer 不能凭空把用户委托到任意合约，前提仍是拿到该 EOA 对相应授权内容的有效签名。

## 一个直观流程

以“首次使用 DEX 时做授权并兑换”为例：

1. 用户的钱包展示将要委托的实现合约、批量调用内容和费用承担方式。
2. 用户签署 ERC-7702 授权，授权中绑定链、实现地址和 nonce。
3. 钱包或 relayer 发送 `0x04` 类型交易，并带上 `authorization_list`。
4. EOA 写入委托标记，委托实现代表该 EOA 执行批量逻辑。
5. 合约可在一笔交易中完成 `approve`、兑换等已明确允许的调用。
6. 后续若要取消委托，可签名授权指向零地址；这会清空 EOA 的代码，但**不会自动清理此前已写入的账户存储**。

## 与 ERC-4337 的关系

ERC-7702 和 ERC-4337 都在改善账户体验，但不是同一个东西：

- **ERC-4337**：以 `UserOperation`、Bundler、EntryPoint、Paymaster 为中心，在协议交易之外构建账户抽象流程。
- **ERC-7702**：直接新增协议层交易类型，让既有 EOA 可以设置代码委托。

两者可以组合。例如，ERC-7702 让用户沿用原 EOA 地址并使用可编程账户逻辑；ERC-4337 的 Paymaster 基础设施仍可为满足规则的操作提供赞助。

结合本仓库已有的 Sponsor 实验，需要特别避免一个误解：**7702 只提供账户可编程性，不会替代 Sponsor 的鉴权、限流、预算、审计和熔断。** 即使有 7702，Paymaster / Sponsor 仍必须把赞助授权绑定到明确的调用、费用、有效期、链和账户上下文。

## 安全重点：委托不是轻量级开关

### 1. 委托实现相当于钱包核心代码

一旦 EOA 委托给实现合约，该实现若能任意调用、转账或升级，风险接近把整个账户控制权交出去。前端不能只显示“开启智能钱包”，而要清楚展示：

- 委托到哪个实现地址；
- 实现是否可升级、升级权限属于谁；
- 该次操作会调用哪些目标合约和函数；
- 委托是否持续生效，以及如何撤销。

### 2. 委托代码必须自行处理重放保护和权限边界

ERC-7702 授权元组防的是“设置同一委托”的重放问题，不等于委托实现里的业务调用天然安全。实现合约应对自身签名、批量调用、会话密钥和管理员操作使用明确的 nonce、域分隔、过期时间、调用白名单和额度限制。

OpenZeppelin 文档特别提醒：实现不当的委托合约可能让攻击者几乎完全控制签名者的 EOA。

### 3. 初始化与存储布局需要预先设计

委托实现运行在 EOA 的存储上下文。若初始化逻辑允许任何人抢先写关键 slot，或新版本与旧存储布局冲突，账户可能被锁死或被夺权。

因此不要把“部署时构造函数初始化”的习惯直接搬过来。应设计一次性、可验证、抗抢跑的初始化流程，并为版本迁移和 storage layout 留出明确规则。

### 4. 撤销委托不等于抹除历史状态

授权到零地址可以去掉 delegation indicator，却不会自动删除 EOA 存储中的数据，也不会撤销此前对第三方 Token 的 allowance。需要把“取消委托”“撤销 token 授权”“清理 / 迁移状态”视为不同动作。

### 5. 不要把 `tx.origin` 当作安全授权依据

ERC-7702 改变了 EOA 的代码执行模型，依赖 `tx.origin` 进行身份或权限判断的旧模式更容易出现错误假设。新实现应以显式签名、`msg.sender` 语义、白名单和可审计的授权数据建立权限边界。

## 对当前 Monad 学习项目的可执行启发

当前不应直接把已有 Sponsor 服务对外开放。更合理的后续实验顺序是：

1. **先确认 Monad Testnet 支持情况**：RPC / 节点是否接受 `0x04` 交易，钱包和开发工具是否能构造 `authorization_list`。
2. **写最小、不可升级的委托执行器**：只支持受限的 `executeBatch`，禁止任意 `delegatecall`，并写失败用例测试。
3. **先做本地与测试网验证**：验证批量调用原子性、链 ID / nonce 重放保护、撤销后行为，以及存储残留的可见性。
4. **再考虑 Sponsor 组合**：仅为白名单操作签发短时赞助授权；分别篡改 sender、callData、gas 参数、有效期并确认都被拒绝。
5. **最后才讨论公开产品化**：需要钱包交互提示、认证、限流、配额、预算计量、日志、监控和熔断器。

## 这次学习的关键理解

ERC-7702 的价值不只是“少点一次确认”。它给了既有 EOA 一个协议层、可选择的可编程入口，可能让账户抽象能力更容易覆盖已有地址和资产。

但它也把风险集中到“用户究竟委托了什么代码”上。对用户来说，授权一个委托实现不应被当作普通登录签名；对开发者来说，安全重点从“能不能批量调用”转向“委托逻辑能否被精确限制、审计、撤销与恢复”。
